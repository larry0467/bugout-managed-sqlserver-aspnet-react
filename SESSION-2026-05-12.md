## What's live in prod

### Tickets page ‚Äî the headline rework
- **Three view modes** via segmented toggle: Table / Board / Calendar. Choice persists per user.
- **Multi-select filters** on the toolbar: Applications, Statuses, Types, Escalation Stages, Developers, Labels. Each has a visible "All X" option in the dropdown so users can clear without hunting for the X icon. Empty selection = no filter.
- **Sort toggle** in the toolbar: Newest submitted first / Oldest submitted first. Persists. Drives Board+Calendar; Table column-sort still wins on header click.
- **Show closed** checkbox honors `IsClosedLike` from the per-org status dictionary, not the literal string "CLOSED".
- **Auto-filter for non-admins**: `DEVELOPER` / `VIEWER` are server-side scoped to tickets where `AssignedTo == callerEmail`. `PLATFORM_OWNER` / `SUPER_ADMIN` bypass. Enforced in `TicketController.GetAll`.

### Kanban Board view
- One column per status from the per-org `TicketStatusDef` dictionary. Drag a card between columns ‚Üí updates status. Closed-like columns hide unless **Show closed** is checked.
- Cards lead with `#ID Title`, render label chips with the label's own color, and show priority / host app / Claude / checklist progress / due date as tags. Overdue tickets glow red.
- **Inline assignee dropdown on every card** ‚Äî always rendered as a clickable Select. Shows the user's display name (not email), placeholder "Unassigned" when empty. Click bubbling is stopped so the dropdown doesn't open the modal or initiate a drag.
- Click anywhere else on the card opens the **detail modal**.

### Detail modal / expanded-row panel (shared renderer)
- Workflow stepper at top (Submitted ‚Üí Super-admin ‚Üí Platform-owner ‚Üí In progress ‚Üí Owner approval ‚Üí Completed).
- **Action row**: Video / Download / Share / Resolve. When no video is attached, shows a `NO VIDEO UPLOADED` chip in place of the video trio. Video modal now actually stops playback when closed (was just CSS-hidden before).
- **Attachments section**: aggregates every file on the ticket ‚Äî purple chip = chat attachment, cyan chip = widget bug-report direct attachment. One click opens the SAS URL in a new tab.
- **Edit bar**: Status / Assignee / Due date / Labels ‚Äî all wired to the same handlers the table column uses, so the modal is fully editable.
- **Tabs**: Details / Chat / Checklist / Activity / Claude Activity.

### Custom statuses
- Per-org `TicketStatusDef` table seeded with the original 7 (OPEN, IN_PROGRESS, IN_REVIEW, READY_FOR_TESTING, VERIFIED, RESOLVED, CLOSED) on first read.
- **Settings ‚Üí Ticket Statuses card** allows reorder, rename, recolor, toggle Closed-like, add new, delete. Refuses delete when a status is still in use by tickets (move them first).
- Status dropdown on tickets, the toolbar Status filter, and the Board columns all read from this dictionary.

### Labels + checklists
- `TicketLabel` table is per-org. UI is the `LabelChips` component (`bugs-managed-admin/src/components/LabelChips.tsx`) ‚Äî colored chips with create+attach popover, preset color palette.
- `TicketChecklistItem` table is per-ticket. `ChecklistPanel` component renders a list with checkboxes + progress bar. Progress rolls up to a chip on the Kanban card.

### Activity feed + @-mentions
- `TicketActivity` rows are written by `TicketActivityLogger` on every meaningful mutation: STATUS_CHANGED, ASSIGNED, LABEL_ADDED/REMOVED, CHECKLIST_ADDED/COMPLETED, DUE_DATE_SET, NOTE_ADDED, MENTIONED, SCREENSHOT_ADDED, CHAT_SCREENSHOT_ADDED.
- Activity tab in the detail panel renders them in a Timeline.
- `@name` or `@email` in a chat note triggers `MENTIONED` activity + Slack DM + Google Chat post + email (via Comms Managed).

### Files in widget + chat
- Widget bug-report form has a **Files drop zone** (renamed from Screenshots). Accepts any file type, paste images into description, drag/drop, or pick. Non-image files render as a paperclip chip with truncated filename.
- Chat input is wrapped in `Upload.Dragger` ‚Äî drop files onto the textarea, or paste images, or click the paperclip button.
- Server cap is **25 MB** per file, with a block-list on executable / script extensions (.exe, .bat, .ps1, .js, .jar, .php, .aspx, .vbs, etc.).
- Stored in a new `screenshots` Azure Blob container, fetched via user-delegation SAS.

### Google Chat integration
- New `Project.GoogleChatWebhookUrl` column + `NotificationService.SendGoogleChatAsync` (same `{"text": "..."}` payload shape as Slack).
- Settings page has a **Google Chat Integration card** alongside Slack.
- Wired into `@`-mention notifications so a mention fans out to Slack AND Google Chat AND email.
- See `GOOGLE_CHAT_APP_SETUP.md` in `tmp-investigate/comms` for the wider Chat app context (Comms side).

### Submitter communication
- The existing CommsManaged bridge already wired: ticket created ‚Üí reporter gets email; assigned ‚Üí reporter gets email; comment in chat ‚Üí reporter gets email (and their reply threads back).
- **Added today**: when a ticket is moved to `RESOLVED` or `CLOSED` via the status dropdown OR the dedicated Resolve endpoint, the submitter now gets the "your issue has been resolved" email. Was previously only firing from the owner-approval Approve flow.

### Video upload reliability (the big one)
- **Root cause** of the "transcript saved, video missing" recurring bug: Kestrel's default `MaxRequestBodySize` is 30 MB. Recordings 30‚Äì50 MB were getting killed by Kestrel before our controller's own (then 50 MB) check ever ran. The widget swallowed the failure as a `console.warn`.
- **Fix**:
  - Kestrel `MaxRequestBodySize` + `FormOptions.MultipartBodyLengthLimit` ‚Üí 200 MB app-wide.
  - Per-endpoint `[RequestSizeLimit]` + `[RequestFormLimits]` on `UploadVideo`.
  - Controller's own cap raised to 200 MB.
  - Widget gains `uploadVideoWithRetry` (3 attempts, 1/2/4s backoff). 4xx responses (except 408/429) abort the retry early.
  - Pre-flight size check ‚Äî >200 MB short-circuits with a clear "exceeds X MB limit" message.
  - On hard failure the widget **stays open** showing `Ticket #N was saved ‚Äî but the video upload failed: <reason>` with `Retry video upload` / `Skip & close` buttons. Recorded blob persists in state across retries.

### Comms notification bridge fixes
- **Bug**: notifications weren't landing in Comms inbox at all.
- **Two causes**:
  1. `entityId` was being sent as `ticket.Id.ToString()` (stringified `long`, e.g. `"23"`) ‚Äî Comms's `SendMessageCommand.EntityId` is `Guid?`. JSON binding rejected the request before FluentValidation ran ‚Üí generic 400.
  2. Terraform creates `comms-managed-workspace-id` and `comms-managed-system-sender-user-id` KV secrets with the literal string `placeholder-replace-via-az-keyvault-secret-set`. If those weren't replaced after provisioning, the placeholder gets POSTed as a Guid, fails `NotEqual(Guid.Empty)`, generic 400.
- **Fix**: `entityId` now always sent as `null`; `Guid.TryParse` guard up front with a clear "set this KV secret" log so the failure mode is obvious instead of silent.

### Other smaller things
- Filter `maxTagCount="responsive"` was looping a ResizeObserver against Board re-renders and visibly shaking the page on filter changes ‚Üí switched to a numeric tag count.
- The duplicate "Tickets" page title is gone (the green nav link is the only label now).
- The "Assignee" dropdown's filter chain was over-aggressive ‚Äî relaxed in three steps:
  1. Dropped the specialty-vs-developerCategory match.
  2. Dropped the role gate (`DEVELOPER` / `PLATFORM_OWNER` / `SUPER_ADMIN`) ‚Äî VIEWERs were invisible.
  3. Dropped the project-scope filter ‚Äî devs with explicit `projectIds` were hidden on cross-project tickets.
- Final eligibility: any team member is assignable. Specialty still shows as a hint after the name (e.g. `Dilpreet Singh ¬∑ FRONTEND`).

## Operational notes

### Deploy
- Push to `master` triggers `.github/workflows/deploy.yml`.
- The `prod` GitHub environment has an approval gate ‚Äî you click "Approve and deploy" in the run UI.
- EF migrations run as the last step of the deploy via `dotnet ef database update`.
- If two deploys queue close together, the second one will hit a **Terraform state-lock** conflict and fail in ~15s. Just rerun the failed one (`gh run rerun <id> --failed`) once the first finishes applying.
- **Static Web App deploy step sometimes times out** at 10 min (it's an Azure SWA infra glitch, not your code). Rerun the failed deploy and it'll succeed.

### Verifying a deploy
```bash
gh run list --workflow=deploy.yml --limit 3
gh run watch <run-id> --exit-status
gh run view <run-id> --log-failed   # if conclusion=failure
```

### Logs in prod
App Insights for the Bug Out Container App is the place. The Comms bridge logs under category `BugsManaged.Api.Services.CommsManagedNotificationService` with the prefix `CommsBridge:`. Activity logger logs everything to App Insights too.
